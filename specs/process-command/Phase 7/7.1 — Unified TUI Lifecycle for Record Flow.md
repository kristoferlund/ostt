# 7.1 вҖ” Unified TUI Lifecycle for Record Flow

## Problem

The `handle_record` flow currently tears down `OsttTui` after transcription, then the
action picker and processing animation each set up and tear down their own independent
terminal sessions. This creates three separate terminal lifecycles in what should be
one continuous TUI flow:

1. `OsttTui` вҖ” recording + transcription animation (cleaned up after transcription)
2. `ActionPicker` вҖ” its own `enable_raw_mode` / `EnterAlternateScreen` / cleanup
3. `TerminalGuard` in `execute_action_with_animation` вҖ” yet another terminal session

The user sees the screen flash between terminal sessions. It's also unnecessary
complexity вҖ” the recording, transcription, picker, and processing are all phases of the
same user interaction.

## Dependencies

- **Phase 4 (4.1)** вҖ” Action picker exists
- **Phase 5 (5.2)** вҖ” Processing flow integrated into `handle_record`
- **Phase 6 (6.1)** вҖ” `execute_action_with_animation` exists

## Objective

Keep `OsttTui` alive through the entire `handle_record` flow: recording вҶ’ transcription
animation вҶ’ action picker вҶ’ processing animation вҶ’ cleanup вҶ’ output. One terminal
session, no teardown/rebuild cycles.

## Specification

### Current flow (what exists today)

```
OsttTui::new()          в”Җв”Җв”Җ raw mode, alternate screen в”Җв”Җв”Җв”җ
  recording loop                                           в”Ӯ
  transcription animation (via tui.render_transcription_animation)
tui.cleanup()           в”Җв”Җв”Җ restore terminal в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

ActionPicker::new()     в”Җв”Җв”Җ raw mode, alternate screen в”Җв”Җв”Җв”җ
  picker event loop                                        в”Ӯ
picker.cleanup()        в”Җв”Җв”Җ restore terminal в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

TerminalGuard::new()    в”Җв”Җв”Җ raw mode, alternate screen в”Җв”Җв”Җв”җ
  processing animation                                     в”Ӯ
guard.cleanup()         в”Җв”Җв”Җ restore terminal в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

output (stdout/clipboard/file)
```

### Target flow (after this spec)

```
OsttTui::new()          в”Җв”Җв”Җ raw mode, alternate screen в”Җв”Җв”Җв”җ
  recording loop                                           в”Ӯ
  transcription animation (tui.render_transcription_animation)
  action picker (tui.render_action_picker)                 в”Ӯ
  processing animation (tui.render_transcription_animation в”Ӯ
                        with "Processing..." label)        в”Ӯ
tui.cleanup()           в”Җв”Җв”Җ restore terminal в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ

output (stdout/clipboard/file)
```

### Changes to `src/recording/ui.rs` (OsttTui)

Add a method to render the action picker through OsttTui's existing terminal:

```rust
/// Renders the action picker UI for one frame.
///
/// Returns `Ok(Some(PickerAction))` if the user made a selection or cancelled,
/// `Ok(None)` if the event loop should continue.
pub fn render_action_picker(
    &mut self,
    actions: &[ProcessAction],
    list_state: &mut ListState,
) -> Result<Option<PickerEvent>, Box<dyn Error>>
```

This uses the same terminal that's already open. The rendering code is extracted from
`ActionPicker::draw()` вҖ” same layout (logo header, bordered list, help footer), same
colors, same highlight style.

`PickerEvent` is a simple enum:

```rust
pub enum PickerEvent {
    Selected(String),  // action ID
    Cancelled,
}
```

### Changes to `src/commands/record.rs` (handle_record)

Move `tui.cleanup()` to after all processing is complete. The processing flow runs
inside the TUI session:

```rust
// After transcription succeeds...
// Do NOT call tui.cleanup() here

// Processing flow (if -p was passed)
let output_text = match process.as_deref() {
    None => text,
    Some("") => {
        // Render action picker through OsttTui
        let mut list_state = ListState::default();
        list_state.select(Some(0));
        let selected_action = loop {
            match tui.render_action_picker(&actions, &mut list_state)? {
                Some(PickerEvent::Selected(id)) => break Some(id),
                Some(PickerEvent::Cancelled) => break None,
                None => continue,
            }
        };
        // ... execute action with animation through OsttTui ...
    }
    Some(id) => {
        // ... execute action with animation through OsttTui ...
    }
};

// NOW clean up the TUI
tui.cleanup()?;

// Output
if let Some(file_path) = output_file { ... }
```

For the processing animation, reuse `tui.render_transcription_animation()` вҖ” it already
renders the `TranscriptionAnimation` through OsttTui's terminal. Just create a new
animation with `set_status_label("Processing...")` and run the same spawn-task +
animation-loop + cancel-poll pattern that `transcribe_recording_with_animation` uses.

### Changes to `src/process/execute.rs`

`execute_action_with_animation` remains available for callers that don't have an
`OsttTui` (specifically `handle_process`, `handle_retry`, `handle_transcribe`). Its
`TerminalGuard` approach is correct for those cases вҖ” they don't go through the
recording TUI.

No changes needed to `execute_action_with_animation` itself. The refactor is only in
`handle_record`.

### Changes to `src/process/picker.rs`

`show_action_picker()` remains available for callers that don't have an `OsttTui`
(specifically `handle_process`). It still manages its own terminal.

The rendering logic inside `ActionPicker::draw()` should be extracted into a standalone
function that takes a `&mut Frame` so it can be reused by both `ActionPicker::draw()` and
`OsttTui::render_action_picker()`:

```rust
/// Renders the action picker UI into the given frame area.
///
/// Shared rendering logic used by both the standalone ActionPicker
/// and OsttTui::render_action_picker().
pub fn render_picker_frame(
    frame: &mut Frame,
    area: Rect,
    actions: &[ProcessAction],
    list_state: &mut ListState,
)
```

### While-we're-here fixes in `handle_record`

Since this spec refactors `handle_record`, fix two pre-existing issues observed during
Phase 5 sessions:

**1. Keywords loading inconsistency** (Phase 5, Sessions 1 & 4):
`transcribe_recording_with_animation` loads keywords by manually reading
`~/.config/ostt/keywords.txt` (lines ~388-402), while the processing code in the same
file uses `KeywordsManager`. Replace the manual file read with `KeywordsManager` for
consistency.

**2. Double config load** (Phase 5, Sessions 3 & 4):
When processing is requested, `OsttConfig::load()` is called twice вҖ” once at the start
of `handle_record` for audio config, and again inside the processing branch for process
actions. Since the refactored flow keeps everything in one function scope, load config
once at the top and reuse it throughout.

### What does NOT change

- `handle_process` вҖ” standalone command, uses `show_action_picker()` and
  `execute_action_with_animation()` with their own terminals. No OsttTui involved.
- `handle_retry`, `handle_transcribe` вҖ” use `execute_action_with_animation()` with its
  own terminal. No OsttTui involved.
- `show_action_picker()` public API вҖ” unchanged, still available for non-OsttTui callers.
- `execute_action_with_animation()` public API вҖ” unchanged.
