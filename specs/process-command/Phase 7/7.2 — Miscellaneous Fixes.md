## Problem

## Specification

### Fix 1: Add `ProvidersConfig` re-export to `config/mod.rs`

**Source:** Phase 1, Session 1 ‚Äî marked "FIX THIS"

`ProvidersConfig` is a public type defined in `src/config/file.rs` (line 194) and used
by `OsttConfig.providers`. Other public config types (`ProcessConfig`, `AudioConfig`,
`ActionDetails`, etc.) are re-exported from `src/config/mod.rs`, but `ProvidersConfig`
is not.

**Fix:** Add `ProvidersConfig` to the existing re-export line in `src/config/mod.rs`:

```rust
pub use file::{AudioConfig, OsttConfig, ProvidersConfig, VisualizationType};
```

### Fix 2: Add `cleaned_up` guard to `HistoryViewer::cleanup()`

**Source:** Phase 4, Session 1 ‚Äî marked "FIX THIS"

`KeywordsViewer` and `ActionPicker` both use a `cleaned_up: bool` field to prevent
double-cleanup in their `cleanup()` methods and `Drop` impls. `HistoryViewer` does not
have this guard ‚Äî its `cleanup()` can be called multiple times (once explicitly, once
from `Drop`). While currently idempotent in practice (the crossterm calls don't panic
on double-call), it's an inconsistency.

**Fix:** Add `cleaned_up: bool` field to `HistoryViewer`, initialize to `false` in
`new()`, check and set in `cleanup()`:

```rust
fn cleanup(&mut self) -> Result<()> {
    if self.cleaned_up {
        return Ok(());
    }
    self.cleaned_up = true;
    // ... existing cleanup code ...
}
```

### Fix 3: Improve bash executor error for non-existent commands

**Source:** Phase 3, Session 1 ‚Äî marked "FIX THIS"

When a user configures a bash action with a non-existent command (e.g., `command = "foobar"`),
the error is:

```
Command exited with status 127:
sh: 1: foobar: not found
```

This is technically correct (sh exits 127) but not user-friendly. The user sees "Command
exited with status 127" when the real problem is "command not found."

**Fix:** In `execute_bash_action`, after detecting a non-zero exit, check for exit code
127 (standard "command not found" code) and format the error differently:

```rust
if !output.status.success() {
    let stderr = String::from_utf8_lossy(&output.stderr);
    let code = output.status.code().unwrap_or(-1);
    if code == 127 {
        anyhow::bail!(
            "Command not found. Make sure the command is installed.\nShell output: {}",
            stderr.trim()
        );
    }
    anyhow::bail!("Command exited with status {code}:\n{}", stderr.trim());
}
```

Update the `nonexistent_command_returns_clear_error` test to check for the improved
error message.

### Fix 4: Add mouse scroll support to action picker

**Source:** Phase 4, Session 1 ‚Äî "We should add mouse support"

The history viewer and keywords viewer both support mouse scroll (scroll up/down to
navigate the list). The action picker only supports keyboard navigation.

**Fix:** Add mouse capture and scroll handling to `ActionPicker`, following the same
pattern as `HistoryViewer`:

1. Import `EnableMouseCapture`, `DisableMouseCapture`, `MouseEvent`, `MouseEventKind`
2. Add `EnableMouseCapture` to the `execute!` call in `ActionPicker::new()`
3. Add `DisableMouseCapture` to the `execute!` call in `ActionPicker::cleanup()`
4. Handle `Event::Mouse` in `ActionPicker::run()`:
   - `MouseEventKind::ScrollUp` ‚Üí `self.list_state.select_previous()`
   - `MouseEventKind::ScrollDown` ‚Üí `self.list_state.select_next()`

No click-to-select needed ‚Äî scroll only, matching the history and keywords viewers.
