# 1.1 — Action Config Types

## Problem

OSTT has no concept of post-processing actions. The config system (`src/config/file.rs`) only knows about `[audio]` and `[providers]` sections. To support configurable processing actions, the TOML parser needs new types for the `[[process.actions]]` structure.

## Objective

1. Define Rust types for processing actions (both bash and AI types) with serde deserialization.
2. Integrate them into the existing `OsttConfig` struct so they're loaded automatically.
3. Provide a function to look up an action by its `id`.

## Specification

### New types in `src/config/file.rs`

Add the following types:

```rust
/// The role of an input message sent to an LLM.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "lowercase")]
pub enum InputRole {
    System,
    User,
}

/// A special source that provides dynamic content at runtime.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(rename_all = "lowercase")]
pub enum InputSource {
    /// The transcribed text from the recording
    Transcription,
    /// The user's configured keyword list
    Keywords,
}

/// The content source for an action input message.
///
/// Uses `#[serde(untagged)]` so that TOML input entries are disambiguated by
/// field name alone — the user writes `source = "transcription"`, `file = "~/prompt.txt"`,
/// or `content = "literal text"` and serde matches the correct variant.
///
/// Variant order defines precedence: if a user specifies multiple fields on the
/// same input (e.g. both `source` and `content`), serde picks the first matching
/// variant silently. Precedence: `source` > `file` > `content`.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(untagged)]
pub enum InputContent {
    /// Dynamic source: transcription or keywords
    Source { source: InputSource },
    /// Path to a file whose contents become the message content
    File { file: String },
    /// Literal text content
    Literal { content: String },
}

/// A single input entry for an AI action.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ActionInput {
    /// Message role: "system" or "user" — enforced by `InputRole` enum.
    pub role: InputRole,
    /// The content source — exactly one of: literal content, a special source, or a file path.
    /// Enforced at the type level via `InputContent` enum.
    #[serde(flatten)]
    pub input_content: InputContent,
}

/// Type-specific fields for a processing action.
///
/// Uses `#[serde(tag = "type")]` so the TOML `type` field drives which variant
/// serde expects. Required fields for each variant are enforced at deserialization
/// time — no runtime validation needed.
#[derive(Debug, Clone, Serialize, Deserialize)]
#[serde(tag = "type", rename_all = "lowercase")]
pub enum ActionDetails {
    /// A bash command that receives transcription via stdin
    Bash {
        /// Shell command to execute
        command: String,
    },
    /// An AI chat completion action
    Ai {
        /// Provider/model string (e.g. "openai/gpt-4o")
        model: String,
        /// Input messages for the LLM
        inputs: Vec<ActionInput>,
    },
}

/// A single processing action.
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ProcessAction {
    /// Unique identifier for this action (used in CLI: `-p clean`)
    pub id: String,
    /// Human-readable display name (shown in action picker)
    pub name: String,
    /// Action type and its type-specific configuration.
    /// The TOML `type` field ("bash" or "ai") determines which fields are required.
    #[serde(flatten)]
    pub details: ActionDetails,
}

/// Top-level process configuration.
#[derive(Debug, Clone, Default, Serialize, Deserialize)]
pub struct ProcessConfig {
    /// List of configured processing actions
    #[serde(default)]
    pub actions: Vec<ProcessAction>,
}
```

### Integration with OsttConfig

Add an optional `process` field to `OsttConfig`:

```rust
pub struct OsttConfig {
    pub audio: AudioConfig,
    #[serde(default)]
    pub providers: ProvidersConfig,
    #[serde(default)]
    pub process: ProcessConfig,
}
```

### Lookup function

Add a method to `ProcessConfig`:

```rust
impl ProcessConfig {
    /// Finds an action by its `id` field.
    pub fn get_action(&self, id: &str) -> Option<&ProcessAction> {
        self.actions.iter().find(|a| a.id == id)
    }
}
```

### Validation

The following constraints are enforced at the type level by serde deserialization — no runtime validation needed:

- **Bash actions must have `command`** — it's a required field on `ActionDetails::Bash`
- **AI actions must have `model`** — it's a required field on `ActionDetails::Ai`
- **AI actions must have `inputs`** — it's a required field on `ActionDetails::Ai`
- **`source` values must be `"transcription"` or `"keywords"`** — enforced by `InputSource` enum
- **`role` values must be `"system"` or `"user"`** — enforced by `InputRole` enum
- **Each input has exactly one content source** — enforced by `InputContent` enum
- **`type` must be `"bash"` or `"ai"`** — enforced by `ActionDetails` enum

The only remaining runtime validation:

Add a `validate` method to `ProcessAction` that checks:
- AI actions must have **at least one** input (serde accepts `inputs = []`)

Call validation during config load and return a clear error if invalid.

### Re-exports

Export the new types from `src/config/mod.rs`:

```rust
pub use file::{ProcessConfig, ProcessAction, ActionDetails, ActionInput, InputContent, InputRole, InputSource};
```

## Acceptance Criteria

### Valid configurations (must deserialize successfully)

- [ ] A complete bash action (`id`, `name`, `type = "bash"`, `command`) deserializes correctly
- [ ] A complete AI action (`id`, `name`, `type = "ai"`, `model`, `inputs` with entries) deserializes correctly
- [ ] A config with both bash and AI actions in the same `[[process.actions]]` list deserializes correctly
- [ ] A config file with no `[process]` section still loads (defaults to empty actions list)
- [ ] An AI input with `role = "system"` and `content = "..."` deserializes correctly
- [ ] An AI input with `role = "user"` and `content = "..."` deserializes correctly
- [ ] An AI input with `source = "transcription"` deserializes correctly
- [ ] An AI input with `source = "keywords"` deserializes correctly
- [ ] An AI input with `file = "~/path.txt"` deserializes correctly

### Invalid ProcessAction configurations (must fail deserialization)

- [ ] Missing `type` field entirely
- [ ] Unknown `type = "python"`
- [ ] Missing `id` field
- [ ] Missing `name` field
- [ ] Bash action missing `command`
- [ ] AI action missing `model`
- [ ] AI action missing `inputs` key entirely
- [ ] AI action missing both `model` and `inputs`

### Invalid ActionInput configurations (must fail deserialization)

- [ ] Missing `role` field
- [ ] Unknown `role = "admin"`
- [ ] No content field at all (no `content`, `source`, or `file`)
- [ ] Unknown `source = "bogus"`

### Edge cases

- [ ] AI action with empty `inputs = []` deserializes but is rejected by `validate()` at config load
- [ ] An input with multiple content fields (e.g. both `source` and `content`) silently uses the highest-precedence field (`source` > `file` > `content`)

### Lookup

- [ ] `ProcessConfig::get_action("clean")` returns the matching action
- [ ] `ProcessConfig::get_action("nonexistent")` returns `None`

### Build

- [ ] `cargo check` passes
- [ ] `cargo clippy -- -D warnings` passes
- [ ] `cargo test` passes
