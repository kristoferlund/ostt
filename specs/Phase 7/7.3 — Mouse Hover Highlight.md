# 7.3 — Mouse Hover Highlight

## Problem

The history viewer and action picker support mouse scroll and click, but there's no
visual feedback showing which list item the mouse cursor is hovering over. The user
moves the mouse and nothing happens visually until they click or scroll.

## Dependencies

- **7.2** — Action picker mouse support must be in place first (Fix 4 in Miscellaneous Fixes)

## Objective

Add a subtle hover highlight to list items on the history screen and the action picker.
When the mouse cursor is over a list row, that row gets a very faint background tint —
barely visible dark gray on the black background.

Not added to the keywords screen — clicking a keyword doesn't do anything, so hover
feedback would be misleading.

## Specification

### Hover color

Use the same highlight color that the keywords viewer uses for its keyboard selection:

```rust
const HIGHLIGHT_BG: Color = Color::Rgb(20, 20, 20);
```

This is already defined in all three viewers. For hover, use a subtler value:

```rust
const HOVER_BG: Color = Color::Rgb(10, 10, 10);
```

This is barely visible against the `BG` of `Color::Rgb(0, 0, 0)` — just enough to
indicate "this row is under your cursor." The keyboard-selected row keeps its existing
`HIGHLIGHT_BG` which is brighter.

If a row is both hovered AND selected, the selected style wins (no blending needed).

### Implementation approach

Ratatui does not have built-in hover support. The approach is:

1. **Track mouse position** — store `hovered_index: Option<usize>` on the viewer struct.
2. **Update on `MouseEventKind::Moved`** — compute which list row the mouse Y coordinate
   maps to, accounting for the list area position, borders, and scroll offset.
3. **Clear on mouse leave** — set `hovered_index = None` when the mouse moves outside the
   list area.
4. **Apply hover style in rendering** — when building `ListItem`s, check if the item's
   index matches `hovered_index` and apply `HOVER_BG` as the item's style. The `List`
   widget's `highlight_style` (for keyboard selection) takes precedence over item-level
   styles, so the selected row stays at `HIGHLIGHT_BG`.

### Hit-testing: mouse Y to list index

The list area has a known `Rect` (from the layout split). Given:
- `list_inner_y = list_area.y + 1` (1 for top border)
- `row_height` — items in history are 2 lines each (timestamp + text), items in picker
  are 1 line each
- `scroll_offset = list_state.offset()`

```rust
fn mouse_y_to_index(&self, mouse_y: u16, list_area: Rect, row_height: u16) -> Option<usize> {
    let inner_top = list_area.y + 1; // border
    let inner_bottom = list_area.y + list_area.height - 1; // border
    if mouse_y < inner_top || mouse_y >= inner_bottom {
        return None;
    }
    let relative_y = mouse_y - inner_top;
    let visible_index = relative_y as usize / row_height as usize;
    let actual_index = visible_index + self.list_state.offset();
    if actual_index < self.entries.len() {
        Some(actual_index)
    } else {
        None
    }
}
```

The list area `Rect` needs to be stored or recomputed. The simplest approach: store
`list_area: Rect` on the viewer struct, set it during `draw()`, read it during mouse
event handling.

### Changes to `src/history/ui.rs`

1. Add `HOVER_BG` constant: `Color::Rgb(10, 10, 10)`
2. Add `hovered_index: Option<usize>` and `list_area: Rect` fields to `HistoryViewer`
3. In `handle_mouse`, add `MouseEventKind::Moved` arm that computes `hovered_index`
   from mouse Y position. History items are 2 lines tall (timestamp + text).
4. In `draw`, when building list items, set the item style to `HOVER_BG` if its index
   matches `hovered_index`:
   ```rust
   let mut item = ListItem::new(vec![timestamp, text]);
   if Some(i) == self.hovered_index && Some(i) != self.list_state.selected() {
       item = item.style(Style::default().bg(HOVER_BG));
   }
   ```
5. Store the computed `list_area` rect for use in mouse hit-testing.

### Changes to `src/process/picker.rs`

Same approach, but simpler — picker items are 1 line tall.

1. Add `HOVER_BG` constant: `Color::Rgb(10, 10, 10)`
2. Add `hovered_index: Option<usize>` and `list_area: Rect` fields to `ActionPicker`
3. Handle `Event::Mouse` in the `run()` loop (alongside the scroll handling from 7.2):
   - `MouseEventKind::Moved` → compute `hovered_index`
   - `MouseEventKind::ScrollUp/Down` → existing scroll behavior
4. In `draw`, apply `HOVER_BG` to the hovered item (same pattern as history)
5. Store `list_area` rect.

Note: The `render_picker_frame` function (extracted in 7.1) will need access to
`hovered_index` to apply the hover style. Pass it as a parameter.

### What does NOT change

- Keywords viewer — no hover. Mouse scroll still works but no visual hover feedback.
- Recording TUI — no mouse support at all.
