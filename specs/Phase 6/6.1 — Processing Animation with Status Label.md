# 6.1 — Processing Animation with Status Label

## Problem

The existing transcription animation (`TranscriptionAnimation`) shows the OSTT logo sweeping across the screen but has no text indicating what's happening. With two distinct phases (transcription and processing), the user needs to know which phase is active.

## Objective

1. Add a status label to the animation that shows "Transcribing..." or "Processing..." below the logo.
2. Make the label configurable so callers can set it.
3. Integrate the animation into the processing flow (for AI actions, which can take several seconds).

## Specification

### Changes to `src/transcription/animation.rs`

Add a status label field:

```rust
pub struct TranscriptionAnimation {
    // ... existing fields ...
    status_label: String,
}
```

Add a setter:

```rust
impl TranscriptionAnimation {
    pub fn set_status_label(&mut self, label: &str) {
        self.status_label = label.to_string();
    }
}
```

In the `draw` method, render the status label centered below the logo animation:

```rust
// After rendering the logo characters...
let label_x = (width / 2).saturating_sub(self.status_label.len() as u16 / 2);
let label_y = center_y + 2; // 2 lines below the logo center
if !self.status_label.is_empty() && label_y < height {
    frame.buffer_mut().set_string(
        area.x + label_x,
        area.y + label_y,
        &self.status_label,
        Style::default().fg(Color::Rgb(128, 128, 128)), // gray
    );
}
```

Default label: `"Transcribing..."` (set in `new()`).

### Integration with record command

In `src/commands/record.rs`, in the `transcribe_recording_with_animation` function, set the label before the animation loop:

```rust
animation.set_status_label("Transcribing...");
```

### Processing animation in commands

For the processing phase (in `handle_record`, `handle_process`, etc.), reuse the same animation with a different label:

```rust
animation.set_status_label("Processing...");
```

The animation loop pattern for processing should be extracted into a shared helper in `src/process/mod.rs` or `src/process/execute.rs`:

```rust
/// Executes an action with an animated progress indicator.
///
/// Shows the OSTT logo animation with a "Processing..." label while the action runs.
/// Cancellable via Esc/q/Ctrl+C.
pub async fn execute_action_with_animation(
    action: &ProcessAction,
    transcription: &str,
    keywords: &[String],
) -> anyhow::Result<Option<String>>
```

This function:
1. Sets up a fresh terminal (raw mode, alternate screen)
2. Spawns the action execution as a tokio task
3. Renders the animation with "Processing..." label
4. Polls for cancel input (Esc/q/Ctrl+C)
5. Returns `Ok(Some(result))` on success, `Ok(None)` on cancel, `Err` on failure
6. Restores terminal on exit

For bash actions (which are typically fast), still show the animation — it provides consistent UX and handles the case where a bash command takes a while.

## Acceptance Criteria

- [ ] During transcription, "Transcribing..." appears below the logo
- [ ] During processing, "Processing..." appears below the logo
- [ ] The label is centered horizontally
- [ ] The label uses a gray color (distinguishable from the white logo)
- [ ] `execute_action_with_animation` shows the animation during action execution
- [ ] Pressing Esc/q during processing cancels the action
- [ ] Terminal is restored after animation ends (both success and cancel)
- [ ] `cargo check` passes
