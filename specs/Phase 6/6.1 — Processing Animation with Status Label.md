# 6.1 — Processing Animation with Status Label

## Problem

The existing transcription animation (`TranscriptionAnimation`) shows the OSTT logo sweeping across the screen but has no text indicating what's happening. With two distinct phases (transcription and processing), the user needs to know which phase is active.

## Dependencies

- **Phase 3 (3.2)** — `process::execute_action` dispatcher (called by `execute_action_with_animation`)
- **Phase 5** — `handle_process`, `handle_record`, `handle_retry`, `handle_transcribe` (callers of the animation helper)

## Objective

1. Add a status label to the animation that shows "Transcribing..." or "Processing..." below the logo.
2. Make the label configurable so callers can set it.
3. Integrate the animation into the processing flow (for AI actions, which can take several seconds).

## Specification

### Changes to `src/transcription/animation.rs`

Add a `status_label` field to `TranscriptionAnimation`:

```rust
pub struct TranscriptionAnimation {
    // ... existing fields (chars, phase, start_time, min_duration, frame_count) ...
    status_label: String,
}
```

Initialize in `new()`:

```rust
pub fn new(_terminal_width: usize) -> Self {
    Self {
        // ... existing fields ...
        status_label: "Transcribing...".to_string(),
    }
}
```

Add a setter:

```rust
pub fn set_status_label(&mut self, label: &str) {
    self.status_label = label.to_string();
}
```

In the existing `draw` method, add label rendering **after** the character rendering loop
(after the `for anim_char in &self.chars` block). The variable `center_y` is already
defined in `draw()` as `height / 2`:

```rust
// Render status label centered below the logo
let label_x = (width / 2).saturating_sub(self.status_label.len() as u16 / 2);
let label_y = center_y + 2; // 2 lines below the logo center
if !self.status_label.is_empty() && label_y < height {
    frame.buffer_mut().set_string(
        area.x + label_x,
        area.y + label_y,
        &self.status_label,
        Style::default().fg(Color::Rgb(128, 128, 128)), // gray
    );
}
```

Default label: `"Transcribing..."` (set in `new()`). This means existing callers
(`transcribe_recording_with_animation` in `record.rs`) get the label automatically
without code changes. The explicit `set_status_label("Transcribing...")` call in
`record.rs` is optional but recommended for clarity.

### Integration with record command

In `src/commands/record.rs`, in the `transcribe_recording_with_animation` function,
set the label before the animation loop for explicitness:

```rust
animation.set_status_label("Transcribing...");
```

### Processing animation helper: `execute_action_with_animation`

Add a shared helper in `src/process/execute.rs`:

```rust
/// Executes an action with an animated progress indicator.
///
/// Shows the OSTT logo animation with a "Processing..." label while the action runs.
/// Cancellable via Esc/q/Ctrl+C.
///
/// # Returns
/// - `Ok(Some(result))` — action completed successfully
/// - `Ok(None)` — user cancelled during processing
/// - `Err(...)` — action failed
pub async fn execute_action_with_animation(
    action: &ProcessAction,
    transcription: &str,
    keywords: &[String],
) -> anyhow::Result<Option<String>>
```

This function:
1. Sets up a fresh terminal (raw mode, alternate screen, `CrosstermBackend`, `Terminal::new`)
2. Creates a `TranscriptionAnimation` and calls `set_status_label("Processing...")`
3. Spawns the action execution (`execute_action`) as a tokio task
4. Runs an animation loop: render frame, poll for cancel input (Esc/q/Ctrl+C), sleep 50ms, check if task finished
5. Returns `Ok(Some(result))` on success, `Ok(None)` on cancel, `Err` on failure
6. Restores terminal on exit (raw mode off, leave alternate screen, show cursor)
7. Uses a `Drop`-based cleanup guard (same pattern as `HistoryViewer` and the Phase 4 action picker) to ensure terminal is restored even on panic or early return

#### Why not refactor the transcription animation loop?

The transcription animation loop in `record.rs:306-344` renders through `OsttTui`'s
already-open terminal via `tui.render_transcription_animation()`. The processing
animation sets up its own independent terminal session. These use different terminal
owners, so unifying them into a single helper would require either abstracting over
the terminal source or breaking the recording TUI's encapsulation. The duplication is
acceptable — the pattern is the same (spawn task, render loop, poll cancel, sleep) but
the terminal management differs.

### Callers

All commands that chain processing must use `execute_action_with_animation`:

- `src/commands/process.rs` — `handle_process` (Phase 5.1)
- `src/commands/record.rs` — `handle_record` when `-p` is set (Phase 5.2)
- `src/commands/retry.rs` — `handle_retry` when `-p` is set (Phase 5.2)
- `src/commands/transcribe.rs` — `handle_transcribe` when `-p` is set (Phase 5.2)

For bash actions (which are typically fast), still show the animation — it provides
consistent UX and handles the case where a bash command takes a while.
