# 4.1 — Action Picker TUI

## Problem

When the user enables processing without specifying an action ID, they need a way to select which action to run from the configured list. This should be a ratatui TUI consistent with OSTT's existing UI style.

## Objective

1. Build a fullscreen ratatui-based action picker that lists configured actions and lets the user select one.
2. Return the selected action's ID, or indicate cancellation.

## Specification

### New module: `src/process/picker.rs`

```rust
/// Result of the action picker interaction.
pub enum PickerResult {
    /// User selected an action
    Selected(String),  // action ID
    /// User cancelled (Esc/q)
    Cancelled,
}

/// Shows the action picker TUI and waits for user selection.
///
/// # Arguments
/// * `actions` - List of available actions to display
///
/// # Returns
/// The ID of the selected action, or `Cancelled` if the user presses Esc/q.
///
/// # Errors
/// - If terminal initialization fails
/// - If rendering fails
pub fn show_action_picker(actions: &[ProcessAction]) -> anyhow::Result<PickerResult>
```

### UI layout

The picker must follow the same visual structure as the existing OSTT TUI screens
(`src/history/ui.rs` is the primary reference, `src/keywords/ui.rs` is secondary).

Shared layout pattern to replicate:
1. **Outer padding:** `Block::default().padding(Padding::uniform(1)).style(bg: BG)` fills the full terminal area.
2. **Main block:** unstyled `Block::default()` inside the padding, providing `inner_area`.
3. **Vertical split of `inner_area`:**
   - **Header** (`Constraint::Length(3)`) — the OSTT logo: `" ┏┓┏╋╋ \n ┗┛┛┗┗ \n"`, left-aligned, white on black.
   - **List area** (`Constraint::Min(0)`) — bordered `List` widget with title.
   - **Footer** (`Constraint::Length(1)`) — centered help text in dim gray.

```
┌──────────────────────────────────────┐
│                                      │
│  ┏┓┏╋╋                               │
│  ┗┛┛┗┗                               │
│                                      │
│ ┌ Process action ──────────────────┐ │
│ │ > Clean up text                  │ │
│ │   Generate CLI command           │ │
│ │   Answer question                │ │
│ │   Translate to English           │ │
│ └──────────────────────────────────┘ │
│                                      │
│    ↑/↓ navigate  Enter select        │
│    Esc cancel                        │
│                                      │
└──────────────────────────────────────┘
```

Reuse the same color constants from history/keywords:

```rust
const BG: Color = Color::Rgb(0, 0, 0);
const FG: Color = Color::Rgb(255, 255, 255);
const HIGHLIGHT_BG: Color = Color::Rgb(20, 20, 20);
const HELP_FG: Color = Color::Rgb(100, 100, 100);
```

- Selected item highlighted with `HIGHLIGHT_BG` and `> ` marker (same `highlight_symbol` as history)
- List block: `Block::default().title(" Process action ").borders(Borders::ALL)`
- Help footer: `"↑/↓ select, ↵ confirm, esc/q cancel"`, centered, `HELP_FG` color

### Keyboard controls

| Key | Action |
|-----|--------|
| `↑` / `k` | Move selection up |
| `↓` / `j` | Move selection down |
| `Enter` | Confirm selection |
| `Esc` / `q` | Cancel |

### Edge cases

- If only one action is configured, skip the picker entirely and return that action's ID directly.
- If no actions are configured, return an error "No processing actions configured. Add actions to ~/.config/ostt/ostt.toml".

### Terminal management

Follow the same terminal lifecycle pattern used in `HistoryViewer` and `KeywordsViewer`:

1. **Setup:** `enable_raw_mode()`, `execute!(stdout, EnterAlternateScreen)`, `CrosstermBackend::new(stdout)`, `Terminal::new(backend)`
2. **Event loop:** `event::read()` with key dispatch
3. **Cleanup:** `disable_raw_mode()`, `LeaveAlternateScreen`, `show_cursor()`
4. **Drop impl:** `impl Drop` that calls cleanup, ensuring terminal is restored even on panic/early return

Reference `src/history/ui.rs` for the exact pattern. Mouse capture is not needed for the action picker (keyboard-only is sufficient).
