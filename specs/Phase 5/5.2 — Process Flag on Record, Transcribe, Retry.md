# 5.2 — Process Flag on Record, Transcribe, Retry

## Problem

Processing should also be chainable directly after recording, transcribing, or retrying. The `-p` / `--process` flag enables this without a separate `ostt process` invocation.

## Dependencies

- **5.1** — Process subcommand (establishes the processing flow)
- **Phase 3 (3.2)** — `process::execute_action` dispatcher
- **Phase 4 (4.1)** — `process::picker::show_action_picker` TUI

## Objective

1. Add the `-p` / `--process` optional flag to `record`, `transcribe`, and `retry` commands.
2. When set, chain processing after transcription completes.

## Specification

### CLI changes in `src/app.rs`

Add to `Record`, `Transcribe`, and `Retry` command variants:

```rust
/// Enable processing after transcription. Optionally specify action ID to skip picker.
#[arg(short = 'p', long = "process", value_name = "ACTION", num_args = 0..=1, default_missing_value = "")]
process: Option<String>,
```

The `num_args = 0..=1` and `default_missing_value = ""` attributes make this work as:
- Not present → `None` (no processing)
- `-p` with no value → `Some("")` (show picker)
- `-p clean` → `Some("clean")` (use specific action)

The interpretation in handler code:

```rust
match process.as_deref() {
    None => { /* no processing, output raw transcription */ }
    Some("") => { /* show action picker */ }
    Some(id) => { /* look up action by id */ }
}
```

Also add to the top-level `Cli` struct for the default record command (no explicit `record` subcommand):

```rust
/// Enable processing after transcription
#[arg(short = 'p', long = "process", value_name = "ACTION", num_args = 0..=1, default_missing_value = "")]
process: Option<String>,
```

Note: The `-p` flag on the top-level `Cli` struct and on each subcommand are independent.
When the user writes `ostt -p clean`, it populates `Cli.process`. When they write
`ostt record -p clean`, it populates `Commands::Record.process`. The routing code merges
them the same way it merges `-c` and `-o` today.

### Handler changes

Update `handle_record`, `handle_transcribe`, and `handle_retry` signatures to accept the process parameter:

```rust
pub async fn handle_record(
    clipboard: bool,
    output_file: Option<String>,
    process: Option<String>,
) -> Result<(), anyhow::Error>

pub async fn handle_transcribe(
    file: PathBuf,
    clipboard: bool,
    output_file: Option<String>,
    process: Option<String>,
) -> Result<(), anyhow::Error>

pub async fn handle_retry(
    recording_index: Option<usize>,
    clipboard: bool,
    output_file: Option<String>,
    process: Option<String>,
) -> Result<(), anyhow::Error>
```

### Processing flow (after transcription succeeds)

After transcription succeeds and the transcription text is available:

1. If `process` is `None`, proceed to output as before (no processing).
2. If `process` is `Some("")`, load config, show the action picker. If cancelled, output the raw transcription (fall through to normal output path).
3. If `process` is `Some(id)`, load config, look up the action by id. Error if not found: `"Unknown action '{id}'. Use 'ostt process --list' to see available actions."`.
4. Load keywords via `KeywordsManager`.
5. Execute the action via `process::execute_action`.
6. Save both the raw transcription and the processed result to history. Both are saved as regular transcription entries via `save_transcription` — no special marking or schema distinction. This produces two history entries per run.
7. The **processed result** replaces the raw transcription in the output path (clipboard/file/stdout). The raw transcription is only in history.

### Processing is skipped on transcription failure

If transcription fails (API error, cancelled, no model configured, etc.), the `-p` flag
is silently ignored. No picker is shown, no action is executed. The existing error
handling paths are unchanged.

### Terminal lifecycle for `handle_record`

The `handle_record` function has a complex TUI lifecycle. With processing added, the
sequence becomes:

1. **Recording TUI up** — `OsttTui::new()`, recording loop with visualization
2. **Transcription animation** — reuses the recording TUI's terminal, shows logo animation with "Transcribing..." label (see Phase 7)
3. **Recording TUI down** — `tui.cleanup()`
4. *(If processing requested and transcription succeeded)*
5. **Action picker TUI up** — `show_action_picker()` sets up its own terminal (only if `process` is `Some("")` and there are multiple actions)
6. **Action picker TUI down** — picker restores terminal on return
7. *(If action was selected, not cancelled)*
8. **Processing animation up** — fresh terminal setup, logo animation with "Processing..." label
9. **Processing animation down** — terminal restored
10. **Output** — write result to file/clipboard/stdout (no TUI active)

Each TUI stage manages its own terminal setup and cleanup independently. The recording
TUI is always cleaned up before any processing TUI starts.

For `handle_retry` and `handle_transcribe`, the lifecycle is simpler (no recording TUI):
1. Transcribe (no animation in these commands currently)
2. Action picker (if needed)
3. Processing animation
4. Output

### Routing changes in `src/app.rs`

Update the match arm for `Record`:

```rust
None | Some(Commands::Record { .. }) => {
    let (clipboard, output, process) = match cli.command {
        Some(Commands::Record { clipboard, output, process }) => (clipboard, output, process),
        None => (cli.clipboard, cli.output, cli.process),
        _ => unreachable!(),
    };
    commands::handle_record(clipboard, output, process).await?;
}
```

Update the match arm for `Retry`:

```rust
Some(Commands::Retry { index, clipboard, output, process }) => {
    commands::handle_retry(index, clipboard, output, process).await?;
}
```

Update the match arm for `Transcribe`:

```rust
Some(Commands::Transcribe { file, clipboard, output, process }) => {
    commands::handle_transcribe(file, clipboard, output, process).await?;
}
```

## Acceptance Criteria

- [ ] `ostt -p -c` records, transcribes, shows action picker, processes, copies to clipboard
- [ ] `ostt -p clean -c` records, transcribes, runs "clean" action, copies to clipboard
- [ ] `ostt -c` (no -p) records, transcribes, copies to clipboard (unchanged behavior)
- [ ] `ostt record -p clean -c` same as `ostt -p clean -c` (explicit subcommand)
- [ ] `ostt transcribe file.mp3 -p clean` transcribes and processes
- [ ] `ostt transcribe file.mp3 -p` transcribes, shows picker, processes
- [ ] `ostt retry -p clean` retries and processes
- [ ] `-p nonexistent` returns a clear error with the action name
- [ ] Cancelling the picker outputs the raw transcription
- [ ] Both raw and processed results are saved to history (two entries)
- [ ] Transcription failure silently skips processing (no picker shown)
- [ ] Terminal is properly managed across recording → transcription → picker → processing transitions
- [ ] `cargo check` passes
- [ ] `cargo clippy -- -D warnings` passes
- [ ] `cargo test` passes
