# 5.2 — Process Flag on Record, Transcribe, Retry

## Problem

Processing should also be chainable directly after recording, transcribing, or retrying. The `-p` / `--process` flag enables this without a separate `ostt process` invocation.

## Objective

1. Add the `-p` / `--process` optional flag to `record`, `transcribe`, and `retry` commands.
2. When set, chain processing after transcription completes.

## Specification

### CLI changes in `src/app.rs`

Add to `Record`, `Transcribe`, and `Retry` command variants:

```rust
/// Enable processing after transcription. Optionally specify action ID to skip picker.
#[arg(short = 'p', long = "process", value_name = "ACTION")]
process: Option<Option<String>>,
```

This uses `Option<Option<String>>`:
- Not present → `None` (no processing)
- `-p` with no value → `Some(None)` (show picker)
- `-p clean` → `Some(Some("clean".into()))` (use specific action)

Also add to the top-level `Cli` struct for the default record command:

```rust
/// Enable processing after transcription
#[arg(short = 'p', long = "process", value_name = "ACTION", global = true)]
process: Option<Option<String>>,
```

### Handler changes

Update `handle_record`, `handle_transcribe`, and `handle_retry` signatures to accept the process parameter:

```rust
pub async fn handle_record(
    clipboard: bool,
    output_file: Option<String>,
    process: Option<Option<String>>,
) -> Result<(), anyhow::Error>
```

After transcription succeeds and before output:

1. If `process` is `None`, proceed to output as before.
2. If `process` is `Some(None)`, show the action picker. If cancelled, output the raw transcription.
3. If `process` is `Some(Some(id))`, look up the action. Error if not found.
4. Execute the action.
5. Save both the raw transcription and the processed result to history.
6. The processed result replaces the raw transcription in the output path (clipboard/file/stdout).

### Record command specifics

In `handle_record`, the processing step happens after the TUI is cleaned up, same as the current output logic. The action picker needs its own terminal setup since the recording TUI has been torn down.

For the animation during processing (AI actions can take several seconds), use the existing `TranscriptionAnimation` rendered in a fresh terminal session. This is addressed in Phase 7 (animation with status label). For now, processing runs without animation — the terminal will appear to hang briefly during the API call. This is acceptable as a first pass.

### Routing changes in `src/app.rs`

Update the match arm for `Record`:

```rust
None | Some(Commands::Record { .. }) => {
    let (clipboard, output, process) = match cli.command {
        Some(Commands::Record { clipboard, output, process }) => (clipboard, output, process),
        None => (cli.clipboard, cli.output, cli.process),
        _ => unreachable!(),
    };
    commands::handle_record(clipboard, output, process).await?;
}
```

Similarly for `Retry` and `Transcribe`.

## Acceptance Criteria

- [ ] `ostt -p -c` records, transcribes, shows action picker, processes, copies to clipboard
- [ ] `ostt -p clean -c` records, transcribes, runs "clean" action, copies to clipboard
- [ ] `ostt -c` (no -p) records, transcribes, copies to clipboard (no processing, unchanged behavior)
- [ ] `ostt transcribe file.mp3 -p clean` transcribes and processes
- [ ] `ostt retry -p clean` retries and processes
- [ ] `-p nonexistent` returns a clear error
- [ ] Cancelling the picker outputs the raw transcription
- [ ] Both raw and processed results are saved to history
- [ ] `cargo check` passes
- [ ] `cargo clippy -- -D warnings` passes
- [ ] `cargo test` passes
