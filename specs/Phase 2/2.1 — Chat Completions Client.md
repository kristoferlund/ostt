# 2.1 — Chat Completions Client

## Problem

OSTT only has API clients for audio transcription (multipart file upload to `/v1/audio/transcriptions`). AI processing actions need a chat completions client that sends a messages array to `/v1/chat/completions` and returns the assistant's response text.

## Objective

1. Implement a chat completions API client that works with OpenAI-compatible endpoints.
2. Parse the `provider/model` format from action config to determine API key and model name.
3. Support all providers that offer OpenAI-compatible chat endpoints (OpenAI, Groq, DeepInfra, Berget).

## Specification

### New module: `src/process/chat.rs`

```rust
/// Configuration for a chat completions request.
pub struct ChatConfig {
    /// The provider ID (e.g., "openai", "groq")
    pub provider_id: String,
    /// The model name to send in the API request (e.g., "gpt-4o")
    pub model: String,
    /// The API key for authentication
    pub api_key: String,
    /// The endpoint URL
    pub endpoint: String,
}

/// A single message in the chat completions request.
pub struct ChatMessage {
    pub role: String,
    pub content: String,
}

/// Sends a chat completions request and returns the assistant's response text.
pub async fn chat_completions(
    config: &ChatConfig,
    messages: &[ChatMessage],
) -> anyhow::Result<String>
```

### Provider/model parsing

Add a helper function:

```rust
/// Parses "provider/model" into (provider_id, model_name).
/// Returns error if the format is invalid.
pub fn parse_model_spec(spec: &str) -> anyhow::Result<(String, String)>
```

Example: `"openai/gpt-4o"` → `("openai", "gpt-4o")`

### Endpoint mapping

Map provider IDs to their chat completions endpoints:

| Provider | Endpoint |
|----------|----------|
| `openai` | `https://api.openai.com/v1/chat/completions` |
| `groq` | `https://api.groq.com/openai/v1/chat/completions` |
| `deepinfra` | `https://api.deepinfra.com/v1/openai/chat/completions` |
| `berget` | `https://api.berget.ai/v1/chat/completions` |

Unknown providers should return an error with a message listing supported providers.

### API request format

Standard OpenAI chat completions format:

```json
{
  "model": "gpt-4o",
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."}
  ]
}
```

### Response parsing

Parse the standard response format and extract `choices[0].message.content`. Handle:
- Empty choices array
- Missing content field
- HTTP error codes (401, 403, 429, 5xx) with human-readable messages consistent with the existing transcription API error style

### Error handling

Follow the same pattern as `src/transcription/api/openai.rs`:
- Connection errors → "Failed to connect to {provider} API server."
- Timeout → "Request to {provider} timed out."
- 401 → "API key is invalid or expired. Please run 'ostt auth'."
- 429 → "Rate limit hit. Please wait and try again."
- 5xx → "API server experiencing issues. Please try again later."
