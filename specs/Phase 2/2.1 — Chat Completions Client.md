# 2.1 — AI CLI Tool Executor

## Problem

OSTT needs to send transcription text to an AI model for post-processing (cleanup, translation, summarization, etc.). Rather than implementing API clients for each provider, OSTT delegates to external CLI tools that the user already has installed and configured. This gives access to a wide range of models and providers with no API client code to maintain.

## Objective

1. Extend the config types to support a `tool` field on AI actions, specifying which CLI tool to invoke.
2. Implement an executor that invokes the selected CLI tool in non-interactive mode, pipes the resolved prompt, and captures the response text.
3. Support four CLI tools: OpenCode, Claude Code, Gemini CLI, and Codex CLI.

## Specification

```rust
pub enum ActionDetails {

        command: String,

    Ai {
        /// Which CLI tool to invoke

```

### New module: `src/process/ai.rs`

```rust
/// Executes an AI action by invoking the specified CLI tool in non-interactive mode.
///
/// Constructs a prompt from the resolved messages, pipes it to the tool via stdin,
/// and captures stdout as the result.
///
/// # Arguments
/// * `tool` - Which CLI tool to invoke
/// * `model` - Model identifier (passed through to the tool as-is)
/// * `messages` - Resolved input messages (from input resolution)
/// * `tool_binary` - Optional binary path override (defaults to standard binary name)
/// * `tool_args` - Optional extra CLI arguments appended after required ones
///
/// # Errors
/// - If the CLI tool is not found on PATH
/// - If the tool exits with non-zero status
/// - If the tool times out
pub async fn execute_ai_action(
    tool: &AiTool,
    model: &str,
    messages: &[ResolvedMessage],
    tool_binary: Option<&str>,
    tool_args: Option<&[String]>,
) -> anyhow::Result<String>
```

### Prompt construction

Messages are split into two parts: a **system prompt** (all system messages concatenated)
and a **user prompt** (all user messages concatenated).

Multiple messages of the same role are concatenated with blank line separators.

How these are passed to the tool depends on the tool:

- **OpenCode**: No system prompt separation — system and user prompts are concatenated
  together (with `[System]` / `[User]` role labels) and piped to stdin as one string.
- **Claude Code**: System prompt passed via `--system-prompt` flag. User prompt piped to stdin.
- **Gemini CLI**: System prompt passed as the `-p` argument. User prompt piped to stdin.
- **Codex CLI**: System prompt passed as the positional argument to `codex exec`. User prompt piped to stdin.

For OpenCode (the only tool without system prompt separation), the concatenation format is:

```
[System]
{system message content}

[User]
{user message content}
```

### CLI tool invocation

All tools are invoked using `tokio::process::Command` with `Stdio::piped()` for stdin,
stdout, and stderr. The prompt is written directly to the child process's stdin pipe —
**no shell interpolation or escaping is involved**. This is binary-safe and handles
special characters, newlines, unicode, emoji, and arbitrary-length content without issues.

```rust
// Determine binary: use tool_binary override, or default for the tool
let binary = tool_binary.unwrap_or(tool.default_binary());

// Build required args for this tool (model, system prompt, etc.)
let mut args = tool.build_required_args(model, &system_prompt);

// Append user-provided extra args, if any
if let Some(extra) = tool_args {
    args.extend(extra.iter().cloned());
}

let mut child = Command::new(binary)
    .args(&args)
    .stdin(Stdio::piped())
    .stdout(Stdio::piped())
    .stderr(Stdio::piped())
    .spawn()?;

let mut stdin = child.stdin.take().unwrap();
stdin.write_all(user_prompt.as_bytes()).await?;
drop(stdin); // close stdin to signal EOF

let output = child.wait_with_output().await?;
```

The `tool_binary` override replaces the default binary name (e.g., `"claude"` → `"/usr/local/bin/claude"`).
The `tool_args` are appended *after* all required arguments, so they cannot interfere with the core invocation.

Each tool uses different binary names, arguments, and stdin content.

All four tools produce clean text on stdout when stdout is piped (not a TTY).
Metadata, progress, and debug info go to stderr. This has been empirically verified.

#### OpenCode

- Binary: `opencode`
- Args: `["run", "--model", "{model}"]`
- Stdin: full concatenated prompt (system + user messages)
- Model format: passed through as-is (e.g., `anthropic/claude-haiku-4-5`, `openai/gpt-5.4-nano`)
- Stdout: clean response text only
- Stderr: ANSI-colored model header (e.g., `> build · claude-haiku-4-5`)

#### Claude Code

- Binary: `claude`
- Args: `["-p", "--system-prompt", "{system_prompt}", "--model", "{model}"]`
- Stdin: user messages only (system messages passed via `--system-prompt` flag)
- Model format: alias (`haiku`, `sonnet`, `opus`) or full name (`claude-haiku-4-5`)
- Stdout: clean response text only
- Stderr: empty
- Notes: `-p` defaults to text output format. Flags like `--tools ""`, `--no-session-persistence`, and `--dangerously-skip-permissions` are unnecessary — we are not executing commands, just processing text.

#### Gemini CLI

- Binary: `gemini`
- Args: `["-p", "{system_prompt}", "-m", "{model}"]`
- Stdin: user messages only (system/instruction passed as the `-p` argument)
- Model format: model name (e.g., `gemini-2.5-flash`, `gemini-2.5-pro`)
- Note: `-p` requires a string argument (it's not a boolean flag). The `-p` value and stdin content are concatenated by Gemini CLI ("Appended to input on stdin if any"). This allows passing the system prompt as the `-p` argument and piping user content via stdin.
- Stdout: clean response text only
- Stderr: empty

#### Codex CLI

- Binary: `codex`
- Args: `["exec", "{system_prompt}", "--model", "{model}"]`
- Stdin: user messages only (system/instruction passed as the positional argument to `codex exec`; Codex wraps piped stdin in `<stdin>` tags and appends it to the instruction)
- Model format: model name as the provider expects it (e.g., `gpt-5.4`, `o4-mini`). Note: available models depend on the user's Codex authentication method (API key vs ChatGPT account).
- `--model` flag selects the model
- Stdout: clean response text only (when piped)
- Stderr: full metadata (model info, session ID, token count, echoed prompt)
- Notes: `--ephemeral` is optional — omitting it causes session files to be saved, which is harmless.

### Timeout

Add a 120-second timeout for all CLI tool invocations. If exceeded, kill the process and return an error.

### Error handling

- Tool not found on PATH: "CLI tool '{tool_name}' not found. Please install it and ensure it's on your PATH."
- Non-zero exit: "AI tool '{tool_name}' failed (exit code {code}):\n{stderr}"
- Timeout: "AI tool '{tool_name}' timed out after 120 seconds"
- Empty stdout: "AI tool '{tool_name}' returned no output"

## Acceptance Criteria

- [ ] `AiTool` enum deserializes all four variants (`opencode`, `claude-code`, `gemini-cli`, `codex-cli`)
- [ ] Deserialization rejects unknown tool values (e.g., `tool = "vim"`)
- [ ] `ActionDetails::Ai` requires `tool` field — missing `tool` fails deserialization
- [ ] `tool_binary` and `tool_args` are optional — omitting them deserializes to `None`
- [ ] `tool_binary = "/custom/path"` overrides the default binary
- [ ] `tool_args = ["--flag", "value"]` deserializes correctly
- [ ] Existing Phase 1 tests are updated to include the `tool` field and still pass

### Prompt construction

- [ ] System and user messages are separated correctly
- [ ] System prompt passed via `--system-prompt` for Claude Code, `-p` arg for Gemini CLI, positional arg for Codex CLI
- [ ] All messages concatenated with role labels for OpenCode (no system prompt separation)
- [ ] Multiple messages of the same role are concatenated with blank line separators

### Tool invocation

- [ ] OpenCode: invoked with `["run", "--model", "{model}"]`, full prompt piped via stdin
- [ ] Claude Code: invoked with `["-p", "--system-prompt", "{system}", "--model", "{model}"]`, user prompt piped via stdin
- [ ] Gemini CLI: invoked with `["-p", "{system}", "-m", "{model}"]`, user prompt piped via stdin
- [ ] Codex CLI: invoked with `["exec", "{system}", "--model", "{model}"]`, user prompt piped via stdin
- [ ] `tool_binary` override changes the spawned binary
- [ ] `tool_args` are appended after required args

### Error handling

- [ ] Missing CLI tool returns a clear error with the tool name
- [ ] Non-zero exit returns error with stderr content
- [ ] Timeout after 120 seconds kills the process and returns error
- [ ] Empty stdout returns a clear error
